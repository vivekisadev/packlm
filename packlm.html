// PACKLM V2 — WHAT CHANGED

Three changes. Always fewer tokens.

v1 had a flaw: for already-compact formats like CSV, adding an alias prefix to every row cost more than it saved. v2 fixes the formula from the ground up.

CHANGE 1
No alias prefix on rows (single entity)
When a file has only one entity type, the row alias is wasted. Every row gets a prefix token that adds nothing. v2 removes it for single-entity files.
R Alice 20 A Delhi ← +1 token per row
Alice 20 A Delhi ← saved
CHANGE 2
Auto enum literals for repeated strings
If a column has 2–10 unique string values, v2 auto-generates single-char codes. Bangalore → B. Saves 1–3 tokens per cell across every row.
Delhi Mumbai Bangalore Delhi ← 4 tokens each
$city D=Delhi M=Mumbai B=Bangalore
D M B D ← 1 token each
CHANGE 3
BPE-accurate token estimator
v1's counter didn't split on commas, quotes, and brackets correctly, making CSV look cheaper than it is. v2 counts punctuation as individual tokens — the way real BPE tokenizers work.
Old: CSV "Alice,20,A" = 1 word + 2 commas = 3
v2: CSV "Alice,20,A" = 5 tokens (each , = 1)
UNCHANGED
Multi-entity files still use aliases
When a file mixes students and schools, aliases like @ST and @SC are necessary for the LLM to know which row belongs to which entity. This is kept in v2.
@ST name age grade
@SC school_name city
ST Alice 20 A
SC "Delhi Public" Delhi

// FORMAT SPECIFICATION v2.0

Seven rules. No exceptions.

v2 adds two new constructs (implicit schema and enum literals) and makes one rule optional (row alias). Everything else is identical to v1.

SyntaxMeaningExample
@ f1 f2 f3NEW v2: Schema without alias. Use when file has ONE entity type. Rows have no prefix — saves 1 token per row.@ name age grade
@ALIAS f1 f2Schema with alias. Required when mixing multiple entity types in one file. Rows must start with ALIAS.@ST name age
@ALIAS f1:i f2:fType hints — :i int, :f float, :b bool. Decoder casts automatically.@ST name age:i gpa:f
$field c=val ...NEW v2: Enum literal. Maps short codes to full values for a field. Placed after schema, before data rows.$city D=Delhi M=Mumbai
val1 val2 val3Data row (single entity — no alias prefix). Values in schema order, separated by spaces.Alice 20 A
ALIAS v1 v2 v3Data row (multi entity — alias prefix required). Used when file contains @ALIAS schemas.ST Alice 20 A
~Null / missing value. Decoded as None / null.Alice ~ A
"val with spaces"Quoted string. Required when value contains whitespace."Alice Smith" 20 A
# commentFull-line comment. Ignored by parsers.# students table
v2 complete example — single entity + enum literals
# PackLM v2.0 — Students with enum literals

@ student_name age:i grade gpa:f city active:b
$city D=Delhi M=Mumbai B=Bangalore C=Chennai H=Hyderabad

"Alice Smith"   20  A   3.9  D  true
"Bob Kumar"     19  B   3.2  M  true
"Charlie Roy"   21  A   3.8  D  true
"Diana Mehta"   20  B+  3.5  B  false
"Ethan Patel"   22  A+  4.0  C  true

# city column: D decodes to Delhi, M to Mumbai, etc.
# Without enums: "Bangalore" = 2-3 tokens. With enum: "B" = 1 token.
# For 100 rows: saves ~200 tokens on city column alone.

System prompt for your LLM (v2): "Data is in PackLM v2 format. A line starting with @ is a schema. If it has an alias (@ST field1 field2), data rows start with that alias. If no alias (@ field1 field2), rows are plain values. Lines starting with $field define enum codes: D=Delhi M=Mumbai means D decodes to Delhi. ~ = null. Quoted values contain spaces."
