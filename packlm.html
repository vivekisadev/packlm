// HOW IT WORKS

Schema once. Values only.

JSON wastes tokens by repeating key names on every single row. PackLM separates the schema from the data — like a database table header — so keys are declared once regardless of how many rows follow.

🔍
The JSON problem
JSON repeats every key name per object. With 5 fields and 100 rows, you spend 500 key-tokens (plus 500 colons, 500 commas, 1000 quotes) on structure alone — wasted context window.
📋
Declare schema once
A line beginning with @ALIAS declares the fields for that entity type. @ST name age grade — appears once, no matter how many student rows follow.
⚡
Values-only rows
Each data row is just the alias plus values in schema order — no keys, brackets or colons. ST Alice 20 A. Multi-word values use quotes. Null is ~.
🧠
LLMs understand it
Add one sentence to your system prompt. LLMs grasp column-oriented tabular data natively — the pattern matches CSV, SQL tables, and spreadsheets in their training data.
// TOKEN BREAKDOWN

Where the savings come from

JSON — token cost per row
{"student_name": "Alice Smith", "age": 20, "grade": "A"}

  "student_name"    → 4 tokens   (quote + student + _ + name + quote)
  :                 → 1 token
  "Alice Smith"     → 3 tokens
  ,                 → 1 token    (repeated between every field)
  {  }  [  ]        → structural overhead on every row

  Per-row cost:  ~18 tokens for just 3 fields
  100 rows:      ~1,800 tokens  (most of it structural noise)
PackLM — token cost
@ST student_name age grade   ← schema: ~5 tokens, paid ONCE

ST "Alice Smith" 20 A         ← row:    ~5 tokens

  100 rows:  5 + (100 × 5) = ~505 tokens
  Saving vs JSON: ~1,295 tokens  (~72% reduction)
  Saving grows linearly with row count

// FORMAT SPECIFICATION

PackLM v1.0

A plain-text, line-oriented format. UTF-8 encoded. Files use the .packlm extension. Designed to be human-readable and LLM-friendly simultaneously.

SyntaxDescriptionExample
@ALIAS f1 f2 f3 Schema definition. Declares field names for entity type ALIAS. Must appear before any data rows for that alias. One schema per entity type. @ST name age grade
ALIAS v1 v2 v3 Data row. Values in the same order as the schema. Alias must match a previously declared schema. ST Alice 20 A
~ Null / missing value. Decoded as None in Python, null in JSON. ST Bob ~ B
"" Explicitly empty string. Different from ~ (null). ST Charlie "" A
"value with spaces" Multi-word string. Required when value contains whitespace. Use \" to escape a literal quote inside. ST "Alice Smith" 20 A
# comment text Line comment. Entire line is ignored by the parser. Useful for labelling entity sections. # student records
(blank line) Ignored by the parser. Use freely for visual separation between entity type groups.

System prompt to add to your LLM calls
You receive data in PackLM format. Lines starting with @ define schema: @ALIAS field1 field2 ... Other lines are data rows: ALIAS val1 val2 ... — values match schema order. ~ means null. Multi-word values are quoted with double quotes.

Write SPEC.md — the formal language definition

        
This is what makes PackLM a real open format rather than just a Python library. A clear spec means developers can implement encoders/decoders in JavaScript, Rust, Go, etc. without asking you anything. Include EBNF grammar, all edge cases, and a version number.

        

          
SPEC.md skeleton

          
# PackLM Format Specification v1.0

## Grammar (EBNF)
document    = line*
line        = schema-line | data-line | comment | blank
schema-line = "@" alias SP field (SP field)* NL
data-line   = alias SP value (SP value)* NL
value       = bare-value | quoted-value | null
null        = "~"
quoted-value = '"' char* '"'
bare-value  = (any char except whitespace, '"', "~")+
comment     = "#" (any char)* NL

## Rules
1. Schema must be declared before its data rows
2. Row field count must match schema field count exactly
3. ALIAS is case-sensitive; convention: 1-3 uppercase letters
4. Files SHOULD use the .packlm extension
5. Encoding: UTF-8 required
6. Line endings: LF or CRLF both valid

## Versioning
Add a version comment at the top of each file:
  # packlm:1.0

## Reference implementation
packlm.py (Python 3.8+) — see github.com/you/packlm
